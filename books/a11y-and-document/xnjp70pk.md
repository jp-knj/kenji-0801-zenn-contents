## ウェブアクセシビリティ入門：一時的な修正のアプローチ
アクセシビリティを緊急に取り組む必要がありますか。
この記事は、最も重要な問題を対処するための手短な概要と参照点を提供します。

### アクセシビリティとは？
新たにアクセシビリティを学ぶ場合、まず基本的な理解を得ることが役立ちます。
- ウェブアクセシビリティの導入：主要な概念、理由、リソースを紹介します。
- ウェブアクセシビリティの視点：アクセシビリティ機能とその全ての人々への利点を紹介する短い非技術的なビデオです。

### 問題の特定
既にアクセシビリティの問題を知っている場合、このセクションをスキップしてもよいです。
- Web アクセシビリティの簡単なチェック：技術的なスキルやアクセシビリティの知識に関係なく、誰でも実施できる簡単なチェックです。

### 修正のスコープを考慮する
すぐに修正したい部分と、後の段階で対処する部分を決定するため、修正のスコープを考慮してください。

### アクセシビリティの目標レベルを設定
一般的に受け入れられているアクセシビリティの目標は、Web Content Accessibility Guidelines (WCAG) Level AA の最新バージョンです。

### 効率的な修正のためのヒント
- あなたのチームの異なるスキルを活用する
- チーム全体で要件を伝える
- 早い段階で解決策を確認する
- ツールを最適化する

### 長期的な計画と管理
いくつかの最も重要なウェブアクセシビリティの問題を解決した後、未来のデザインと開発のプロセス全体にアクセシビリティを統合する計画を立てることが重要です。「ウェブアクセシビリティの計画と管理」ガイドがその計画の策定をサポートします。

この記事は、ウェブアクセシビリティに関する基本的なガイダンスとリソースの紹介を目的としています。より詳しい情報や具体的なガイドラインは、関連リンクや資料を通じて取得できます。


Web アクセシビリティの導入

ウェブアクセシビリティの努力は、組織の文化、プロセス、実践に基づいて確立する必要があります。組織のアプローチとアクセシビリティを整合させ、明確で計測可能な目標を開発・伝達し、組織全体での理解とサポートを確保するためにステークホルダーを関与させます。

### 基本を理解する
アクセシビリティが何であり、なぜ重要なのかを理解しましょう。あなたや同僚がアクセシビリティをよく理解すれば、組織内でより効果的に実施・推進できます。

### ウェブアクセシビリティとは？
ウェブアクセシビリティは、地理、言語、障害の障壁に関係なく、すべての人々に情報とコミュニケーションを提供することをウェブの基本的な約束としています。

### 現状を探る
組織のアクセシビリティの現状や義務を調査し、今後の作業の範囲を決定します。

### 目標を設定
明確な目標により、主要な納品物を特定し、タイムラインを確立し、成功を測定する方法を定義します。

### ビジネスケースを開発
強力なビジネスケースは、ステークホルダーからの支持を得るのに役立ち、アクセシビリティを組織の優先事項とできます。

### 認識を高める
組織全体でアクセシビリティの目標、価値、重要性を伝えます。

### サポートを集める
組織全体のサポートは、アクセシビリティが組織全体に配布され、持続されることを保証するために不可欠です。

本記事では、ウェブサイトのアクセシビリティが「Web Content Accessibility Guidelines (WCAG)」にどのように準拠しているかを判断するための「Website Accessibility Conformance Evaluation Methodology (WCAG-EM)」について紹介します。

### WCAG-EMとは？
WCAG-EM は、ウェブサイトが WCAG にどれだけ準拠しているかを判断するためのアプローチです。WCAG-EM は WCAG のサポートリソースであり、WCAG の追加要件を定義するものではありません。W3C Working Group Note として公開されています。

### 評価の範囲
WCAG-EM は、すべてのウェブサイト、ウェブアプリケーション、モバイルウェブサイトに適用されます。それは、自己評価や第三者評価など、さまざまな状況をカバーしています。

### WCAG-EMの対象者
WCAG-EM は、ウェブサイトの監査のための共通手順を求めるすべての人々のためのものです。それは、内部評価者、外部監査人、ベンチマーカー、研究者に直接使用されるものです。

### WCAG-EMの内容
WCAG-EM は、方法論の使用に関するガイダンスと特定の状況に対する考慮事項を提供します。評価手順は 5 つの主要なステップで詳細に説明されています。

### WCAG-EM レポートツール
WCAG-EM Report Tool は、WCAG-EM のステップに従って評価報告を生成するツールです。

### WCAG-EMの開発者
WCAG-EM は、WCAG 2.0 評価方法論タスクフォース（Eval TF）によって開発されています。これは、Web Content Accessibility Guidelines Working Group（WCAG WG）と Evaluation and Repair Tools Working Group（ERT WG）の共同タスクフォースです。

## 評価手順
このセクションでは、評価手順の段階と活動を説明します。これらの段階は必ずしも連続的ではありません。また、評価の段階で行われる活動の正確な順序は、Web サイトのタイプ、評価の目的、および評価者が使用するプロセスによって異なります。これらの活動の一部は重複できるか、または並行して実行される可能性があります。以下の図は、このセクションで定義された段階間の反復を示しています：

上記のワークフロー図は、5 つの連続するステップを示しています。
1. 評価の範囲を定義する; 
2. 対象となるウェブサイトを探索する
3. 代表的なサンプルを選択する
4. 選択されたサンプルを監査する

## 評価範囲の定義
方法論要件 1.a、方法論要件 1.b、および方法論要件 1.c に従って評価の範囲を定義し、オプションで方法論要件 1.d も定義します。

このステップでは、評価の全体的な範囲が定義されます。これは評価手順の後続のステップに影響を与える基本的なステップです。評価の範囲についての共通の期待を確保するため、評価を依頼した者（Web サイトの所有者であるかもしれないし、そうでないかもしれません）と協議しながら行うのが理想的です。このステップ中の初期の Web サイトの探索は、Web サイトの詳細や必要な評価をよりよく知るために必要かもしれません。Web サイトの詳細な探索はステップ 2: 対象 Web サイトの探索で実施されます。

### Web サイトの範囲を定義
#### 方法論要件1.a:
評価の範囲内の Web ページと Web ページの状態がそれぞれ一意であることが明確であるように、対象となる Web サイトを「適用範囲」に従って定義します。

このステップでは、評価の範囲内の Web ページと Web ページの状態（ターゲット Web サイト）が定義されます。Web サイトのこの範囲は、「適用範囲」のセクションで定められた条件に従って定義されます。

評価者、評価を依頼した者、および結果の評価レポートの読者の間の期待の後での不一致を避けるために、Web ページがその範囲内であることが一意であるようにターゲット Web サイトを定義することが重要です。可能な場所で正規表現や Web アドレス（URI）のリストを使用することをお勧めします。

また、その識別をサポートするためのターゲット Web サイトの特定の側面を文書化することも重要です。これには以下が含まれます：

- サードパーティのコンテンツとサービスの使用;
- Web サイトのモバイル版と言語版;
- 特にそれらがそのように容易に識別できない場合、Web サイトの一部、たとえば、異なる Web アドレスを持つオンラインショップがターゲット Web サイトの一部として考慮される場合。

### ステップ1.b: 適合目標を定義
#### 方法論要件1.b:
評価のためのターゲットとして WCAG 2.0 の適合レベル（"A"、"AA"、または"AAA"）を選択します。

評価プロセスを開始する一部として、評価のためのターゲットとして WCAG 2.0 の適合レベル（"A"、"AA"、または"AAA"）を定義することが含まれます。WCAG 2.0 レベル AA は、一般的に受け入れられており、推奨されるターゲットです。

注： Web サイトの適合目標を超えて評価することは、そのアクセシビリティのパフォーマンスについてより完全な絵を得るためにしばしば有用です。たとえば、Web サイトが特定の適合レベルを完全に満たしていない場合でも、それはより高い適合レベルからの個々の要件を満たす場合があります。この情報を持っていることは、将来の改善をより効果的に計画するのに役立ちます。

### ステップ1.c: アクセシビリティサポート基準を定義
#### 方法論要件1.c:
Web サイトが提供する機能がアクセシビリティをサポートするための Web ブラウザ、支援技術、および他のユーザーエージェントを定義します。

特に新しい Web 技術に関しては、メディアプレーヤーの「キャプションを表示」機能など、Web サイトで提供されるすべてのアクセシビリティ機能が、オペレーティングシステム、Web ブラウザ、支援技術、および他のユーザーエージェントのすべての可能な組み合わせでサポートされるわけではありません。WCAG 2.0 は、Web サイトの特定の文脈、つまり、コンテンツを作成するために使用される言語、Web 技術、および現在利用可能なユーザーエージェントに応じて、サポートする必要がある

### ステップ1.d: 追加の評価要件を定義（オプション）
#### 方法論要件1.d:
評価者と評価を依頼した者が合意した任意の追加の評価要件を定義します。

評価を依頼した者は、ターゲット Web サイトが WCAG 2.0 に適合する範囲を評価するために必要な基準を超えて、追加の情報に興味を持っている場合があります。具体的に、評価を依頼した者が考慮したい点は以下の通りです：

- ターゲット Web サイトから代表的なサンプルを取得する基準を超えて、追加の Web ページの評価。
- ターゲット Web サイト上の問題のタイプの代表的な例だけでなく、すべての問題の発生についての報告。
- ターゲット Web サイトの利用に関する特定のシナリオ、状況、およびユーザーグループの詳細な分析。
- 評価の基準を超えて特定された問題の解決策に関する提案や説明。
- 障害を持つユーザーの観点を取り入れた評価。
- 特定の文書や報告のフォーマットの遵守。

評価者と合意したこれらの追加の評価要件は、評価の初期段階で明確にし、文書化する必要があります。この詳細は、たとえば、サンプルの選択方法や取得方法など、評価結果のレポートにも反映することが期待されます。
