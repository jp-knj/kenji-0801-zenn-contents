## はじめに
### WCAG-EMの存在とその目的
評価と向上を目指すガイドラインとして、WCAG-EM が存在します。この記事では、これらのガイドラインと方法論について詳しく解説します。
## WCAG-EMの基本
### WCAG-EM とは 
**WCAG-EM**は、Web コンテンツのアクセシビリティを評価するための手順を示す方法論です。これは、WCAG(Web Content Accessibility Guidelines)の標準に基づいており、アクセシビリティガイドラインを深く理解した上で適用することが推奨されています。
Web アクセシビリティの評価と向上を目指すための基本的な概念を提供するガイドラインです。特に大規模な Web コンテンツのアクセシビリティ向上を前提として策定されていますが、コンポーネント単位やページ単位での改善にも効果的です。

## WCAG-EM の評価ステップ
WCAG-EM の評価は、以下の 5 つの主要なステップから成り立っています。

1. **スコープの定義**：評価の対象となる Web コンテンツ、評価の目標、および WCAG の適合レベル（conformance level）である A、AA や AAA を決定します。
2. **Web コンテンツの調査**：Web サイト内の主要なページや頻繁に使用されるページ、さまざまなコンテンツの種類を特定します。
3. **評価対象コンテンツの選定**：すべてのページを評価するのは難しいため、代表的なページやランダムに選ばれたページを評価の対象とします。
4. **コンテンツの評価**：選択されたページが WCAG の基準に準拠しているかを確認します。
5. **評価結果の報告**：評価の結果をまとめ、サイトのアクセシビリティレベルを示す報告を作成します。

Web アクセシビリティの評価と向上は、すべてのユーザーにとって使いやすい Web サイトを提供するための鍵となります。WCAG-EM を適切に活用することで、その目標に一歩近づくことができます。

## スコープの定義
全体的な評価対象を定義する基本的なステップです。このステップは、評価手順の後続のステップに大きく影響します。WCAG-EM は大規模な　Web　コンテンツのアクセシビリティを向上させることを目的としていますが、コンポーネントやページごとでも適用できます。

### 適合目標の定義
WCAG 2.0 の適合レベルである A、AA や AAA を評価の目標として定義します。一般的には、WCAG 2.0 レベル AA が推奨されています。

### アクセシビリティサポートの基準を定義
動作することが期待される OS、ブラウザ、AS(Assistive Technology)、やユーザーエージェントの組み合わせを策定します。

**アクセシビリティサポートの基準の具体例**

|---------|---------|-----------|
| Windows | Chrome  | NVDA      |
| Windows | Firefox | JAWS      |
| macOS   | Safari  | VoiceOver |
| Android | Chrome  | TalkBack  |
| iOS     | Safari  | VoiceOver |

## Web コンテンツの調査
このステップでは、Web コンテンツを調査します。その使用方法、目的、機能についての理解を深めます。特に開発チームの外部にとっては、Web サイトの全体像を把握するのは難しいでしょう。評価者が対象の Web サイトについてさらに学ぶことで、後のステップで洗練されます。

- Web サイトの全ての部分にアクセスできる必要があります。特定のエリアにアクセス制限がある場合は、権限は前もって、もらっておきましょう。
- 評価の初期段階で、Web ページの基本的なアクセシビリティをチェックします。この段階でのチェックは、後の詳細な評価のための基盤を築く役割を果たします。
- 色のコントラスト、文書構造やナビゲーションの一貫性など、問題点や改善点をメモしておきます。

### Webサイトの構造と内容の特定
- 主要なページやセクションを特定します。これにはホームページや、ヘッダー、ナビゲーション、フッターからアクセスできる主要なページが含まれます。
- 主要な機能を特定します。これには、商品の購入機能やユーザー登録機能のような主要な機能が含まれます。全ての機能を網羅する必要はなく、Web サイトの主要な目的や目標に関連する機能を中心に特定します。
  - 事例 1: オンラインショッピングを考えてみましょう。まず、いろいろな商品やその価格を見て、気に入ったものを選びます。次に、その商品をカートに入れ、配送先や支払い方法を決めて注文します。 
  - 事例 2: 新しいサイトでアカウントを作成する際、まず「私はロボットではありません」というテストをパスする必要があります。このテストをクリアした後、ユーザー名やパスワードなどの必要な情報を入力する登録フォームに進むことができます。最後に、入力した情報を確認して、アカウントの登録を完了します。
- Web ページの種類やスタイルの多様性を特定します。異なるデザインや機能を持つページは、アクセシビリティのサポートが異なる可能性があるため、これらのページを特定することが重要です。
  - ニュースサイトの記事ページ： 特徴： 多くのテキスト、画像、動画、関連記事へのリンクなどが含まれる。
  アクセシビリティの課題： 視覚障害のあるユーザーが画像の内容を理解するための代替テキストの提供、動画の字幕や音声解説の提供、テキストのコントラストの確保など。
  - E コマースサイトの商品ページ： 特徴： 商品の画像、詳細説明、価格、レビュー、カートに追加するボタンなどが含まれる。 
  アクセシビリティの課題： カートに追加するボタンの大きさや位置の最適化、商品の詳細説明の読みやすさ、レビューのフォントサイズの調整や色のコントラストの確保など。 
  - SNS のプロフィールページ： 特徴： ユーザーの写真、自己紹介文、フォロワー数、投稿一覧などが表示される。 
  アクセシビリティの課題： ユーザーの写真に対する代替テキストの提供、リンクやボタンの明確な識別、投稿の文字サイズや背景色の調整など。
- Web サイトが依存している技術を特定します。HTML, CSS, JavaScript などの基本的な技術や、WAI-ARIA や SVG などの特定の技術が評価の対象となります。

以上のステップを通じて、Web サイトのアクセシビリティを効果的に評価するための準備を整えることができました。

### 評価対象コンテンツの選定
このステップでは、評価対象の Web サイトを代表する Web ページや Web ページの状態のサンプルを選択します。この選択の目的は、評価結果が Web サイトのアクセシビリティの実際のパフォーマンスを適切に反映することを確認するためです。

### 構造化されたサンプルの含める
Web サイトの各部分を代表する Web ページや Web ページの状態を選択します。これには、Web サイトの共通ページ、主要な機能、ページの種類、使用されている Web 技術などが含まれます。

### ランダムに選択されたサンプルを含める
ランダムに選択された Web ページや Web ページの状態のサンプルは、前のステップで選択された構造化されたサンプルが Web サイトの内容を代表しているか確認する指標として機能します。

サンプルの選択は、Web サイトのサイズ、複雑さ、一貫性、開発プロセスの遵守など、多くの要因に依存します。このステップを通じて、Web サイトのアクセシビリティの評価を効果的に行うためのサンプルを選択できます。

## コンテンツの評価
このステップでは、選定された Web コンテンツとその状態を詳細に評価します。そして、構造化されたサンプルとランダムに選択されたサンプルを比較します。この評価では定義された目標の適合性レベルでの WCAG 2.0 の 5 つの適合要件に従って行われます。

### WCAG 2.0 の 5 つの適合要件
Web アクセシビリティは、すべての人々が Web コンテンツをアクセスし、利用することを可能にするための重要な概念です。WCAG 2.0 は、Web コンテンツのアクセシビリティを評価するための基準を提供しています。このセクションでは、WCAG 2.0 への適合性に関する要件とその理解について説明します。

#### 適合要件
Web ページが WCAG 2.0 に適合するためには、以下のすべての要件を満たす必要があります：

- **適合レベル**: 以下の適合レベルのいずれかが完全に満たされていること。
  - **レベルA**: 最低限の適合。Web ページがすべてのレベル A の成功基準を満たすこと
  - **レベルAA**: Web ページがレベル A とレベル AA の成功基準の両方を満たすこと
  - **レベルAAA**: Web ページがレベル A、レベル AA、およびレベル AAA の成功基準のすべてを満たすか、レベル AAA の適合する代替バージョンが提供される。
- **ページ全体**: 適合性（および適合レベル）は完全な Web ページのみに適用され、ページの一部が除外される場合は適合性を達成することはできません。
- **シナリオ全体**: Web ページがプロセス（活動を達成するために完了する必要がある一連のステップ）を提示する一連の Web ページの一部である場合、プロセス内のすべての Web ページが指定されたレベルまたはそれ以上で適合する必要があります。
- **アクセシビリティサポートされた方法**: 成功基準を満たすために、技術のアクセシビリティサポートされた方法のみが頼りにされます。
- **非干渉**: 技術がアクセシビリティサポートされていない方法で使用される場合、または非適合な方法で使用される場合、それらはユーザーがページの残りの部分にアクセスする能力をブロックしてはいけません。

ACT ルールは、Web アクセシビリティの標準をどのようにテストするかを明確にするためのものです。これにより、アクセシビリティのテストが一貫性を持ち、より正確に行われることが期待されます。ACT ルールを理解し、適切に適用することで、ウェブコンテンツやサービスがすべてのユーザーにとってアクセス可能であることを確認できます。

#### Webページ単位の確認
選定されたサンプル内の各 Web ページとページの状態は、目標の適合性レベルでの WCAG 2.0 の 5 つの適合要件に基づいて確認されます。この確認は、関数の起動、データの入力、またはプロセスの開始なしに行われ、Web ページやページの状態のすべてのコンポーネントをカバーします。

#### シナリオ単位の確認
各完全なプロセスについて、手順に従って、ページとページの状態の各インタラクションが目標の適合性レベルでの WCAG 2.0 の 5 つの適合要件に従っているか確認します。

#### 構造化されたサンプルとランダムサンプルの比較
ランダムに選択されたサンプルが、構造化されたサンプルに存在しない新しいタイプのコンテンツや新しい評価結果を示さないことを確認します。もしランダムサンプルが新しいタイプのコンテンツや新しい評価結果を示す場合、構造化されたサンプルが Web サイトの提供するコンテンツを適切に代表していないことを示しています。

WCAG 2.0 の適合要件に関する深い理解と、必要な専門知識が求められます。この記事は、Web アクセシビリティの評価を効果的に行うためのガイドラインとして提供されています。

ACT ルールは、このアクセシビリティの標準をどのようにテストするかを説明するものです。この記事では、ACT ルールの詳細とその重要性について説明します。

### ACTルールとは
ACT ルールは、Web Content Accessibility Guidelines (WCAG) や WAI-ARIA などのアクセシビリティ標準の適合性をテストする方法を説明します。これにより、アクセシビリティテスターは、他のテスターやテストツール、方法論と一貫した方法でエッジケースを評価する方法を知ることができます。ACT ルールは情報提供のみのものであり、WCAG や ARIA の適合性を判断するための必須条件ではありません。

#### ACTルールの特徴
1. **情報提供のみ**: ACT ルールは情報提供のみのものであり、適合性を判断するための必須条件ではありません。WCAG の適合性を判断する基準は、WCAG の成功基準です。
2. **部分的なチェック**: ACT ルールは、特定のアクセシビリティ要件、例えば WCAG 2 の成功基準をチェックします。ACT ルールは、テーブルセルにヘッダーがあるかどうかをテストするかもしれません。
3. **失敗のチェック**: ACT ルールは、WCAG 2 の成功基準や他のアクセシビリティ要件を満たしていない失敗をチェックするように設計されています。
4. **実装**: ACT ルールは、アクセシビリティテストツールや方法論に実装できます。

## 評価結果の報告
評価の結果は、プロセスの最後に報告されますが、検証可能な結果を確保するために、評価プロセス全体を通じてドキュメントされるべきです。ドキュメントは、評価者、評価の依頼者、Web サイトの所有者など、さまざまな関係者に対して異なる機密性レベルを持っています。

#### 評価の各ステップの結果をドキュメントする
各ステップの結果をドキュメントすることは、評価プロセスの透明性を確保し、評価結果の再現性を保証し、この評価に基づいて行われる声明の根拠を提供するために不可欠です。このドキュメントは公開する必要はありませんが、通常、評価の依頼者が機密性のレベルを決定します。

##### 評価の詳細を記録する（オプション）
評価の詳細を記録することは、評価者にとって良い習慣です。これには、評価に使用されたツール、Web ブラウザ、支援技術、その他のソフトウェア、およびそれらを監査するための方法を記録することが含まれます。この記録は通常、内部的に保持され、評価者が別途合意しない限り共有されません。

##### 評価声明を提供する（オプション）
Web サイトの所有者は、この方法論に基づく評価結果に関する公的な声明する場合があります。この声明は、以下の条件が満たされた場合に行うことができます。
1. すべての非オプションの方法論要件が満たされている
2. 定義された適合目標が、監査されたすべての Web ページと Web ページの状態に適合している
3. Web サイトの所有者が、評価声明の有効性と正確性を確保および維持することを約束している

この方法論に従った評価声明には、少なくとも次の情報が含まれます。

- 評価声明が発行された日付
- ガイドラインのタイトル、バージョン、URI
- 評価された適合レベル
- Web サイトの定義
- 依存する Web 技術
- アクセシビリティサポートのベースライン

この方法論に従って、WCAG 2.0 への部分的な適合が達成された場合、評価声明は次の情報も含めることができます。

- WCAG 2.0 に適合していない Web サイトのエリア
- WCAG 2.0 に適合していない理由

以上が、Web アクセシビリティ評価の報告に関するガイドラインの再構成された説明です。この情報を使用して、Web サイトのアクセシビリティを正確に評価し、その結果を適切に報告できます。
