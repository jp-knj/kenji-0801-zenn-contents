## 概要
**WCAG-EM** は、Web サイトがどれだけアクセシビリティの標準、特に WCAG に準拠しているかを評価するための一連の手順を示します。この評価方法は、Web のアクセシビリティを確保するためのガイドラインを詳しく理解した上で適用します。

この評価は主に 5 つの主要なステップから構成されています。
1. **スコープを定義**: どの Web を評価の対象とするか、何を目標として評価するか、そしてどの WCAG のレベル（A、AA、または AAA）に基づいて評価するかを決定します。
2. **Web サイトの調査**: Web サイトの主要なページ、頻繁に使用されるページ、機能や多様なコンテンツの種類などを特定します。
3. **サンプルページの選定**: 全ページを評価することは難しいでしょう、代表的なページやランダムに選ばれたページを評価することが考慮されます。
4. **サンプルページの評価**: 選ばれたサンプルが WCAG の基準に準拠しているかどうかを確認します。
5. **評価結果の報告**: 最後に、評価の結果をまとめて報告し、サイトがどれだけアクセシブルであるかを示します。

この記事では、上記のステップの詳細とともに、WCAG の概念や用語の簡単な説明も提供しています。アクセシビリティは Web の普及にとって不可欠なものであり、これらのガイドラインは、それを確保するための鍵となるものです。

### スコープを定義
- Web サイトの名前
- Web サイトの範囲
- WCAG のバージョン
  - WCAG 2.0, WCAG 2.1
- WCAG のレベル / Conformance target
  - A, AA, AAA
- アクセシビリティの基準
  - OS（Windows、macOS、iOS、Android など）
  - Web ブラウザ（Google Chrome、Firefox、Safari、Edge など）
  - 支援技術（NVDA、JAWS、VoiceOver、TalkBack など）

#### 評価対象のWebコンテンツの特定
評価範囲内の Web ページやその機能・状態を設定する段階です。アクセスする際の Web ページが明確で、目的のサイトが簡単に探せるようにするため、URL のリストの使用が推奨されています。また、評価はモバイル、タブレット、デスクトップなどのデバイスや、言語・地域ごとに異なる場合がありますので、それらの範囲も明確にします。

#### 評価の目標の定義
どの WCAG 2.0 の適合レベルを目指すかを決定します。一般的には、レベル AA が推奨基準とされていますが、全体像を正確に把握するために、レベル AAA を基準にするのも良い選択です。レベル AAA を基準にすると将来的なアクセシビリティの改善を計画するための参考材料にできます。

#### アクセシビリティの基準の設定
アクセシビリティをサポートするための Web ブラウザ、支援技術、その他のユーザーエージェントを定義します。例として、OS、ブラウザ、支援技術の組み合わせを決めます。

このステップでは、評価者は Web サイトが正しく動作することを期待される OS、Web ブラウザ、支援技術、およびその他のユーザーエージェントの組み合わせを決定します。目的のアクセシビリティサポートの共通の期待を一致させるために、協力して定義します。

### Web サイトの調査
対象の Web サイトの詳細を調査します。その中で、サイトの使われ方、目的や主要な機能について理解することが目的です。すべての情報を把握することは難しいでしょう。このステップでは、サイトの全てのページ、機能や使用されている技術のリスト作成を目指します。そして、この調査の成果を元に次のステップで詳しく評価を進めます。簡易チェックは、後程の詳細な評価の時に、特定の問題を見つけ出す手助けとなります。例として、色のコントラストが足りないページや、文書の構造が不適切なページを見つけて、詳細な評価に備えることができます。
- Web 技術
  - HTML, CSS, JavaScript, WAI-ARIA
  - SMIL, SVG, PDF、CMS
  - ライブラリ、フレームワーク
  - 指定された URL
- 主要な機能
  - Web サイトのショップエリアから商品を選択し、購入する
  - 提供されたアンケートを完了し、提出する
  - アカウントを登録する
#### Web サイトの主要なページの特定
対象となる Web サイトの主要なページや、状態を持つページを特定します。多くの場合、ホームページや主要なページは、サイト全体のヘッダーやナビゲーション、フッターからアクセスできます。このステップで、評価対象の主要なページのリストを作成します。

#### Webサイトの中核となる機能を特定
対象となる Web サイトの主要な機能をリストアップします。サイトを探索して、主要な機能を確認します。オンラインショップの購入機能など、一部の機能はすぐに特定できルでしょう。ですが、サブ機能を見つけるのは難しいときもあります。このステップで、利用者にとって重要な機能のリストを作成します。

注： このステップの目的は、すべての機能をリストアップすることではなく、サイトの主要な目的に関連する中心的な機能を特定することです。

#### 様々なWebページのタイプを特定
異なるデザインや機能を持つ Web ページを特定します。これらのページは、異なるデザインテンプレートやスクリプトによって作られ、その内容や外観が異なることがあります。このステップで、評価者は異なるタイプのページやページの状態を特定します。

#### 使用されるWeb技術を特定
Web サイトを動作させるための技術を特定します。基本的な技術（HTML、CSS）から、補助技術（JavaScript、WAI-ARIA）、そして特別な技術（SMIL、SVG、PDF）まで、Web サイトの動作に関連する技術を確認します。

注： 評価結果を報告する際、使用されているコンテンツ管理システムやライブラリ、フレームワークなどの情報も役立ちます。

#### アクセシビリティ関連のページを特定
障害を持つユーザーやアクセシビリティに関連する情報を提供するページを特定します。これには、アクセシビリティに関する情報、ヘルプ、設定方法などの説明が含まれるページもあります。

### サンプルを選択する
このステップでは、評価対象の Web サイトを代表する Web ページとその状態のサンプルを選択します。この選択の目的は、評価結果が合理的な確信を持って Web サイトのアクセシビリティのパフォーマンスを反映することを確保することです。可能な場合、すべての Web ページとその状態を評価することが強く推奨されます。このサンプリング手順はスキップでき、この評価プロセスの残りのステップでの「選択されたサンプル」は Web サイト全体となります。

サンプルの実際のサイズは、多くの要因によって異なります。これには、Web サイトのサイズ、古さ、複雑さ、一貫性、開発プロセスへの順守などの要因が影響します。

このステップの選択は、ステップ 2: 評価対象の Web サイトを探索するで行われた探索に初めて依存します。そして、評価者が対象となる Web サイトの特定の実装の側面についてより多くを学ぶにつれて、次のステップ 4: 選択されたサンプルを監査で継続的に洗練されます。

#### 構造化されたサンプルを含める
すべての特定された一般的な Web ページや必須の機能を反映する Web ページとその状態を選択します。また、Web ページの種類、依存している Web 技術、および他の関連する Web ページも同様に反映されるよう選択します。

以下の内容を含む Web ページとその状態のサンプルを選択します。
- 特定された一般的な Web ページやその状態
- 特定された他の関連する Web ページやその状態
- これらの前のステップで既に反映されていない場合、追加の Web ページやその状態を選択します。

#### ランダムに選択されたサンプルを含める
ランダムに選択された Web ページとその状態のサンプルを選択し、監査に含めます。

ランダムに選択されたサンプルは、前のステップで選択された構造化されたサンプルが Web サイトで提供されるコンテンツを十分に代表しているかどうかを検証する指標として機能します。

ランダムに選択する Web ページとその状態の数は、前のステップで選択された構造化されたサンプルの 10%です。

#### 完全なプロセスを含める
選択されたサンプルには、完全なプロセスを提示するシリーズの一部であるすべての Web ページとその状態が含まれている必要があります。

### 選択したサンプルの監査
定義した目標の適合レベルに従って、ステップ 3 で選択した Web ページのサンプルを監査します。
このステップでは、評価者がステップ 3 で選択した Web ページと Web ページの状態すべてを監査し、構造化されたサンプルとランダムに選択されたサンプルを比較します。この監査は、ステップ 1.b で定義した目標の適合レベルにおける WCAG 2.0 の 5 つの適合要件に従って行われます。

WCAG 2.0 の 5 つの適合要件は次のとおりです。
1. 適合レベル
2. 全ページ
3. 完全なプロセス
4. アクセシビリティをサポートした技術の使用方法のみ
5. 干渉しないこと

これらの適合要件に関する詳しいガイダンスは、本ドキュメントの範囲を超える部分で WCAG 2.0 のガイダンスのレイヤーや適合に関する理解で提供されています。

**注意**: このステップを実行するには、WCAG 2.0 の適合要件と必要な専門知識セクションで述べられている専門知識を深く理解する必要があります。

### すべての初期Webページをチェック
選択したサンプル内で完全なプロセスの中や終わりでない各 Web ページと Web ページの状態が、目標の適合レベルでの WCAG 2.0 の 5 つの適合要件のそれぞれに適合しているかチェックします。

ステップ 3 で選択されたサンプル内で、完全なプロセスの中や終わりでない各 Web ページと Web ページの状態に対して、ステップ 1.b で定義された目標の適合レベルでの WCAG 2.0 の 5 つの適合要件それぞれとの適合を確認します。これには、関数をアクティベートすることなく、データを入力することなく、またはプロセスを開始することなく、Web ページまたは Web ページの状態のすべてのコンポーネントが含まれます。このような機能とインタラクション、完全なプロセスの中や終わりの Web ページや Web ページの状態は、次のステップで評価されます。

**注意**: サンプル内の多くの Web ページと Web ページの状態は、ヘッダー、ナビゲーションバー、検索フォームなど、繰り返し発生するコンポーネントを持っています。要件は完全なページをチェックすることですが、これらのコンポーネントは、異なる外観や動作を示さない限り、各出現ごとに再評価する必要は通常ありません。

#### WCAG 2.0の成功基準
WCAG 2.0 の成功基準が満たされているかどうかを判断する方法は通常いくつかあります。W3C/WAI は WCAG 2.0 の一連の（非規範的な）技術を提供しており、これには特定の WCAG 2.0 の成功基準を満たす方法が文書化されています。これには、コンテンツが特定の WCAG 2.0 の成功基準を満たさないと知られている方法としての一般的な失敗も含まれます。WCAG 2.0 の技術の概念に関するガイダンスは、WCAG 成功基準の理解技術で提供されます。

評価者は、特定の Web コンテンツが WCAG 2.0 の成功基準を満たすか、または満たさないかをチェックするために、このような文書化されたガイダンスを使用できます。文書化された技術と失敗は、評価報告書の背景としても役立ちます。しかし、W3C/WAI によって文書化された特定の技術と失敗セットを使用する必要はありません。実際、評価者は何の技術や失敗も必要としない場合があります。たとえば、評価者は、十分な技術の要件を満たす特定のテスト手順やプロトコルを使用して、WCAG 2.0 の成功基準が満たされているかどうかを評価するかもしれません。これらは公に文書化されているか、または評価者にのみ利用可能である場合があります。技術の使用に関する詳しいガイダンスは、以前にリンクされた WCAG 成功基準の理解技術で提供されています。

**注意**: WCAG 2.0 の成功基準は、各々が「特定の Web コンテンツに適用されるときに真または偽となるテスト可能なステートメント」として定式化されています。特定の成功基準に関連するコンテンツがユーザーに表示されない場合（例：Web ページ上にビデオがない場合）、成功基準は WCAG 2.0 によって「満足」とされます。オプションとして、評価報告書は特定の成功基準に関連するコンテンツがない場合、例えば「存在しない」と明示的に示すことができます。適合に関する理解は、さらなる背景とガイダンスを提供します。

#### 適合する代替バージョン
Web ページまたは Web ページの状態のコンテンツには代替バージョンが存在する場合があります。たとえば、動画コンテンツはキャプション付きとキャプションなしのバージョンで提供される場合があります。WCAG 2.0 の適合代替バージョンの定義に記載されている要件を満たす代替バージョンの助けを借りて、WCAG 2.0 への適合を達成できます。たとえば、キャプションのない動画コンテンツを持つ Web ページは、適合代替バージョンであると認定される動画の代替バージョンを提供することで、WCAG 2.0 を満たすことができます。適合する代替バージョンの理解は、本ドキュメントの範囲を超える部分でさらなるガイダンスを提供します。

**注意**: 代替バージョンは別々の Web ページや Web ページの状態とみなされず、コンテンツの一部とみなされます。Web ページと Web ページの状態は、それらの代替バージョンとともに 1 つのユニット（完全なページ）として評価されます。

#### アクセシビリティサポート
Web ページまたは Web ページの状態のコンテンツは、アクセシビリティをサポートする方法で提供する必要があります（直接または代替バージョンを介して）。たとえば、動画のキャプションは、ユーザーに表示できるように提供する必要があります。WCAG 2.0 のアクセシビリティサポートの定義は、Web コンテンツ技術の使用がアクセシビリティサポートとして資格を得るための特定の要件を定義しています。アクセシビリティサポートの Web 技術の使用の理解は、本ドキュメントの範囲を超える部分でさらなるガイダンスを提供しています。ただし、WCAG 2.0 は、ウェブサイトがアクセシビリティのためにサポートする必要がある特定の閾値やソフトウェアセットを定義していません。そのようなベースラインの定義は、ウェブサイトの目的、対象とする聴衆、および言語など、いくつかのパラメータに依存します。特定のウェブサイトを評価するために使用されるベースラインは、ステップ 1.c で定義されます。

#### 非干渉
Web ページや Web ページの状態のコンテンツが WCAG 2.0 に適合していない場合でも、Web ページや Web ページの状態全体が WCAG 2.0 に適合している可能性があります。たとえば、情報や機能は、アクセシビリティをサポートする代替バージョンを伴って、アシスタント技術によってまだ広くサポートされていない Web コンテンツ技術を使用して、またはアシスタント技術でサポートされていない方法で提供される場合があります。この場合、非適合コンテンツは、Web ページまたは Web ページの状態が WCAG 2.0 に適合できるように、適合コンテンツを否定的に干渉させてはいけません。WCAG 2.0 の非干渉の適合要件は、コンテンツが非干渉として資格を得るための特定の要件を定義しています。要件 5 の理解は、本ドキュメントの範囲を超える部分でさらなるガイダンスを提供しています。

### すべての完全なプロセスをチェック
完全なプロセスに沿った各 Web ページと Web ページの状態すべてのインタラクションが、目標の適合レベルでの WCAG 2.0 の 5 つの適合要件のそれぞれに適合していることを確認します。

ステップ 3.c で特定された完全なプロセスごとに、識別されたデフォルトとブランチのシーケンスの Web ページと Web ページの状態に従い、ステップ 4.a でそれぞれを評価します。ただし、この場合、すべてのコンテンツを評価する必要はなく、プロセス全体で変更されるコンテンツのみを評価します。

このチェックには、特に以下が含まれます。
- フォーム、入力要素、ダイアログボックス、その他の Web ページコンポーネントとのインタラクション。
- 入力の確認、エラーメッセージ、ユーザーインタラクションからの他のフィードバック。
- 異なる設定、好み、デバイス、インタラクションのパラメータを使用した動作。

### 構造化サンプルとランダムサンプルを比較
ランダムに選択されたサンプル内の各 Web ページと Web ページの状態が、構造化されたサンプルに表示されないコンテンツの種類や結果を示さないことを確認します。

構造化サンプルとランダムに選択されたサンプルの間で WCAG 2.0 の成功基準の個々の発生は異なりますが、ランダムに選択されたサンプルは構造化サンプルに存在しない新しいコンテンツの種類を示すべきではありません。また、ランダムに選択されたサンプルの評価結果は、構造化サンプルの結果と新しい結果を示すべきではありません。ランダムに選択されたサンプルが新しいコンテンツの種類や新しい評価結果を示す場合、これは構造化されたサンプルがウェブサイトで提供されているコンテンツを十分に代表していないことを示しています。この場合、評価者はステップ 3 に戻り、新しく特定されたコンテンツの種類と結果を反映する追加の Web ページと Web ページの状態を選択する必要があります。また、ステップ 2 の結果もそれに応じて調整する必要があります。このステップは、構造化されたサンプルがウェブサイトで提供されているコンテンツを適切に代表しているまで繰り返されます。
