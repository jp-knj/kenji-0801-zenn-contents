## はじめに
デザインプロセスは学習と共有を繰り返します。
デザイナーは、自分のアイディアを異なる背景を持つチームメンバーに伝える必要があります。
異なる視点や専門知識を持つ人々を巻き込むことで、より豊かなデザインを生み出すために必要になります。
デザインのコミュニケーションは、理解と共感を構築し、チーム全体で共有されるビジョンを作る上で中心的な役割を果たします。

## デザインのコミュニケーション
### 組織内のステークホルダー
- **プロジェクトマネージャー**：彼らはプロジェクトのスケジュールとユーザーの要件を管理します。彼らとの議論は、デザインの期限と目標を明確にするために必要です
- **開発者**：彼らはデザインの技術的実装を担当します。開発者とのコミュニケーションは、デザインが技術的に実現可能かを確認し、問題解決を助けるために不可欠です
- **デザインマネージャー**：彼らはデザインの品質と方向性を監督します。彼らにはデザインの評価と指導を求めることができます
- **その他の組織ステークホルダー**：データアナリスト、カスタマーサポート、営業など、プロダクトに関わる他の部門の人々も重要な役割を果たします

#### 組織外のステークホルダー
- **組織外の人々**：外部の意見は新しい視点を提供し、製品の改善につながることがあります
- **投資家**：彼らはビジネスの視点からデザインの価値を評価します。投資家に対するプレゼンテーションは、資金調達やサポートを確保するチャンスです
- **クライアント**：彼らは私たちのプロダクトを使う最終的なユーザーの可能性があります。クライアントへのデザインの説明は、彼らのニーズに合わせて調整されるべきです
- **ベンダー**：外部のサービスプロバイダーとのコミュニケーションも、プロダクトの成果に影響を与えます

デザインをコミュニケートするには、これらのステークホルダーのニーズと期待を理解し、適切な言葉とビジュアルを使って伝えることが必要です。
共感を持って接することで、効果的なコミュニケーションが可能になります。そして、これはデザインが成功するための基礎を築くことに他なりません。

## デザインを共有するためのアプローチ
デザインの共有は、聴衆にクリアで明確なメッセージを伝える重要なステップです。

共有する対象と理由の具体例は以下です。
- **他デザイナーにフィードバックを求めるために共有する**
  - 例： 新しいアプリのユーザーインターフェースデザインを共有し、他のデザイナーからの色合いやレイアウトに関するフィードバックを求める。
- **PM/デザインマネージャーに承認を求めるために共有する**
  - 例： プロジェクトマネージャーまたはデザインマネージャーにデザインコンセプトを提出し、プロジェクトの方向性についての承認を得る。
- **開発者に技術的な難易度を評価してもらうために共有する**
  - 例： 開発チームに対してデザインの技術的側面を説明し、実装の実現可能性と必要なリソースを評価してもらう。
- **チームやビジネスのステークホルダーに進捗を示すために共有する**:
  - 例： 定期的なチームミーティングでデザインの進捗状況を共有し、ビジネス目標に対する進行状況を提示

具体的で明示的な言葉を使って説明すること理解をさらに促進できるのではないでしょうか。頭の中に背景、改善策、妥当性の関係性[^1]を整理しておくのも、1 つの手だと捉えています。
![](/images/books/product-design/suggestion.png)

デザインの目的と意図の情報の対称性を保つために、上記の関係を頭の片隅に置いて、3 つのステップを踏んでみましょう。

1. **共有の目的**
2. **文脈の明確化**
3. **解決策を共有**

具体的なデザインの要素や変更点を示すことで、メンバーに対してより具体的なビジュアルイメージを提供できると捉えています。より詳細に深ぼってみます。

### 共有の目的
共有している理由と、対象に何を求めているのかを説明してください。
これを最初から意識しておくと、何が重要かに焦点を当てるのに役立ちます。 その結果、私の要求をよりサポートしてくれる可能性があがります。

#### 明確なリクエストと期待を設定
以下はあいまいな表現となる可能性があります。
- X についてのフィードバックが必要です
- A と B の間でどれを選ぶか、意見をください

そのため、具体的なリクエストをすることが重要です。できないときは、情報が足りてない状態であることを認識し、情報を集めることを意識しています。情報が結晶化するまで、時間をかけても良いと捉えています。
- 例： このデザインの目的は、アプリの利用率を増加させることです。具体的には、ユーザーがアプリを開いたときにすぐに目的の情報にアクセスできるようにします。ユーザーの滞在時間を延ばし、アプリの再訪問率を高めることを目指しています。目標を達成するためのフィードバックを提供してください。

**具体的なリクエストの例**
- デザイン X に関する色の選択やレイアウトについてのフィードバックが欲しいです
- プロジェクト Y の進行状況について、共有しておきます
- A と B のデザインのうち、どちらがユーザーにとって使いやすいでしょうか。アドバイスをいただければ幸いです
- X の作業が完了しました。振り返りとして、結果と学びを共有します

### 文脈の明確化
聴衆からの要求を明確にした後、プロジェクトに関する背景情報を提供することが重要です。

#### 文脈を具体的にして伝える 
- 例： これまでのユーザビリティテストでは、ターゲットユーザーがナビゲーションを理解するのに苦労していました。具体的には、ユーザーは「検索」タブと「設定」タブの違いが明確でなく、目的のページに辿り着くまでに時間がかかっていました。そのため、ナビゲーションの改善に焦点を当ててデザインをリファインしました。

以下の質問を自問することで、関係者にプロジェクトの文脈を提供し、意思決定プロセスや次のステップに対する理解を深めておきましょう。
- 解決しようとしている問題は何でしょうか、なぜその問題を解決するのでしょうか
- プロジェクトのスケジュールと対象範囲は何でしょうか
- 現在のプロジェクトの進捗はどのように進んでいますか
- 関連する場合は、このトピックについて以前に議論されたことはありますか
- 共有する資料の準備状況はどのようになっていますか
- プロジェクトの制約は何でしょうか
- プロジェクトのブラインドスポットは何でしょうか
- 競合他社はこの問題に対してどのようなアプローチをとっていますか

### 解決策を共有
解決策の解像度を上げて、妥当性を測定するにはどうしたら良いかを考えてみましょう。
このフレームワークは、プロダクト開発における意思決定とユーザーのニーズの理解を深めるために設計されています。デザインの意思決定を共有することで、チーム全体の理解を深め、共感を促進し、プロジェクトの成果を最大化することが目的です。

#### 解決策の解像度
具体例： 新しいナビゲーションデザインでは、タブのアイコンとラベルを明確します。各タブの機能をユーザーに明示しています。例えば、「検索」タブには拡大鏡アイコンを配置します。「設定」タブには歯車アイコンを配置します。ユーザーが必要な情報を簡単に見つけられるようにします。アプリの使いやすさを向上させています。

#### 妥当性の測定方法
- ユーザビリティテスト： 実際のユーザーを対象にします。新しいナビゲーションデザインをテストします。ユーザーがタブの機能をどの程度迅速かつ正確に理解できるかを評価します。
- アプリ内アンケート： アプリの使用後、ユーザーに簡単なアンケートを実施します。新しいナビゲーションデザインの使いやすさに関するフィードバックを収集します。
- 使用データの分析： 新しいデザイン導入後のアプリの使用データを分析します。ユーザーのナビゲーションパターンやエラー率の変化を追跡します。
- 目標達成率の追跡： 特定のタスクにおけるユーザーの成功率を追跡します。新しいデザインが目標達成にどのように影響するかを評価します。

これらの方法を通じて、新しいナビゲーションデザインの妥当性を定量的かつ定性的に測定し、改善点を特定してさらに最適化を図ることができます。

1. **「なぜ」このプロダクトを構築しているのか**
    - 解決しようとしている問題や課題を再確認し、プロジェクトの目的や目標を明確にします。

2. **「誰」のために構築しているのか**
    - ユーザー層の特性（年齢、居住地、職業など）を把握し、この機能が彼らのどのようなニーズに対応するかを説明します。

3. **「いつ」と「どこで」使用されるのか**
    - ユーザーがどの段階で、どのような状況下でこの機能を使用するかを具体的に説明します。使用場所（室内、屋外）、トリガーイベント、時間的な制約、使用デバイス、使用状況などを詳細に検討します。

4. **「何」を構築すべきか**
    - 現在の状況や機能の概要を説明し、どのようにして目標を満たしていないかを明確にします。提案された解決策を提示し、その背後にある理由を説明します。また、解決策の長所と短所をリストアップします。

5. **「どのように」測定するか**
    - 提案された解決策が成功したかどうかを判断するための基準や方法を提案します。

6. **次に進む**
    - プロジェクトのタイムライン、予定されているマイルストーン、次のフェーズで達成したい目標などを含めます。具体的な計画やフィードバック収集の方法を示します。例えば、「来週に新しいプロトタイプのテストを行い、その結果を次のチームミーティングで共有する」といった計画を設定します。

いくつかの質問は 1〜2 文で答えても差し支えありません。情報の整理はざっくりと始めて、少しづつ良くすれば良いと捉えています。

##### 具体例
:::details モバイルアプリ開発
以下は、健康管理のための新しいモバイルアプリ開発に関する解決策の共有の例です。
1. **「なぜ」このプロダクトを構築しているのか**: このプロジェクトの目的は、ユーザーが健康管理を簡単に行えるような新しいモバイルアプリを開発することです。ユーザーが自分の健康状態を効果的に追跡し、管理することを容易にすることが目標です。
2. **「誰」のために構築しているのか**: このアプリは、健康意識が高い若年層のユーザーを主なターゲットとしています。このユーザー層は健康とウェルネスに関して積極的で、技術に精通しており、新しいソリューションを受け入れやすい傾向にあります。
3. **「いつ」と「どこで」使用されるのか**: ユーザーは、日常生活のさまざまなシーンでこのアプリを利用します。具体的には、自宅、ジム、オフィスなど、さまざまな場所で健康状態をチェックし、活動を追跡できます。
4. **「何」を構築すべきか**: 現在のアプリは情報が散在しており、ユーザーが必要なデータを見つけにくいという問題があります。この解決策では、すべての健康情報を一元管理するシステムを提供し、ユーザーインターフェースを使いやすくすることで、この問題を解決します。
5. **次のステップ**: この解決策の効果は、利用頻度の増加とユーザーからのポジティブなフィードバックを通じて測定します。具体的には、アプリの使用状況の分析、ユーザーからのアンケート、フォーカスグループを通じて得られたフィードバックをもとに、アプリのパフォーマンスを評価します。得られたフィードバックは、機能改善への適用と、次週のプロジェクトミーティングでの改善案の提示に活用されます。

このようなアプローチを通じて、エンドユーザーのニーズに合わせた効果的なモバイルアプリを開発します。健康管理のための総合的なソリューションを提供することを目指しています。
:::

このような思考のフレームワークを用いることで、エンジニアやデザイナーは、プロダクト開発の目的とユーザーのニーズを深く理解し、より実用的なフィードバックを提供できます。
デザインの意思決定を共有することで、チーム全体の理解と共感を促進する。
プロジェクトの成果を最大化できるでしょう。

## 論理の明確化
フレームワークのステップの 1 つとして、「論理を明確にする」ステップがあり、これはプレゼンテーション以外の場面でも有効です。デザインや意思決定において、感覚や視覚的な好みだけでなく、データやリサーチ、ビジネス目標などの論理を意識的に使用することで、以下のメリットが得られます。

- 自分の決定に疑問を投げかけ、より良い解決策を見つける。
- 信憑性を上げる。
- 不要な話し合いやフィードバックのサイクルを削減し、協力をより効率的にする。

ただし、デザインのダサさや美的感覚を数値化して利益につなげることの難しさもあります。デザイン決定を行うときは、その背後にある論理を自問してみることが重要です。例えば、「私は。.。だと考える」を「データは。.。と示している」「リサーチによると。..」のような形式で言い換えることを検討しましょう。これは反論されにくく、より良い解決策につながる可能性があります。

言い換えが難しいとき、または論理的な根拠がないときは、他者の意見を求めることが推奨されます。

デザイン決定の根拠として使用できる確かな理由のリストは以下の通りです。

- **データと分析**
    - 例：「カスタマーサポートの担当者から、ユーザーが一度のセッションで複数のアイテムを追加できないことに対するクレームが最も多いとの報告があります」

- **リサーチ**
    - 例：「リサーチによれば、メインナビゲーションを「ハンバーガーメニュー」で隠すと、見つけやすさが半減する」

- **ビジュアルデザイン、デザインパターン、ブランドの整合性**
    - 例：「この色は、デザインシステムのカラーパレットから色調が外れています」
    - 例：「これらの項目は、近接しているために関連しているように見えますが、実際には関連していません」

- **技術的な理由（実現可能性、デザインシステムの制限）**
    - 例：「バックエンドの都合上で新しい要素を表示できません」
    - 例：「デザインシステムの UI ライブラリでは、その UI をサポートしていません」

- **プロジェクトの文脈（範囲、タイムライン、リソース、ロードマップ）**
    - 例：「ブランドのイラストを担当している外注は、予算の都合上で利用できません」
    - 例：「次の四半期にローンチ予定の新しい機能と衝突する可能性があります」

このリストは、特定の解決策の利点と欠点を明確にするのに役立ちます。

プロジェクトやデザインの意思決定において、異なる関係者が異なる価値観や優先順位を持っています。例えば、一部の関係者はデータとその結果を重視するかもしれませんが、他の人々はデータプライバシーや持続可能性のような要素を優先する可能性があります。
ですが、徹底的に議論で論理を明確にするよりも、大枠での方向性を示すことを優先したほうが良いと考えています。

メンバーの顔色を見ながら働きかけるよりも、市場やユーザーの反応をみたいと考えるときも来るでしょう。
